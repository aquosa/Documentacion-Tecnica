1 Acerca de este release

Communications Server para Windows proporciona conectividad SNA para sistemas Windows, lo cual permite la conexión a IBM z/OS Communications Server y a otras implementaciones SNA que dan soporte a conexiones LLC2, SDLC, X.25 y Enterprise Extender. Communications Server para Windows también proporciona una interfaz de adaptador abierta para adaptadores OEM.

IBM Communications Server para Windows Versión 6.4 es una versión actualizada de la versión 6.1.3 que proporciona una nueva funcionalidad de ARB progresivo, Retardo de conmutador de vía de acceso, soporte IPv6 para TN3270E, cliente API SNA y soporte Java para CPI-C de Java 1.6. También proporciona las últimas actualizaciones de mantenimiento de las versiones 6.1.2 y 6.1.3. La versión 6.4 es una instalación completa, y si ya hay instalada una versión anterior inferior a 6.1.3, debe desinstalarse la versión anterior antes de instalar la versión 6.4.

Este documento contiene información suplementaria a la ayuda en línea y las publicaciones. Describe las nuevas funciones añadidas, restricciones y correcciones, y ofrece consejos y sugerencias.

Muchas gracias por elegir IBM Communications Server.

2.1 Requisitos de hardware

Communications Server para Windows, versión 6.4, se ejecuta en cualquier sistema operativo de 32 bits soportado por Windows XP, Windows Server 2003 (Standard Edition o Enterprise Edition) o Windows Vista o Windows Server 2008. Dependiendo del entorno de red y la plataforma Windows utilizada, es posible que sean necesarios un procesador más rápido y memoria adicional.

Es necesario disponer de 5 MB de espacio de disco en la unidad de arranque y 175 MB en cualquier unidad de disco duro para uso permanente.

Los clientes API SNA se ejecutan en modalidad de 32 bits en cualquier hardware que dé soporte a Windows XP, Windows Server 2003 (Standard Edition o Enterprise Edition), Windows Vista o Windows Server 2008.Los clientes API SNA requieren 25 MB en cualquier unidad de disco duro para uso permanente.

Los clientes de administración remota se ejecutan en cualquier hardware que dé soporte a sistema operativos de 32 bits para Windows XP, Windows Server 2003 (Standard Edition o Enterprise Edition), Windows Vista o Windows Server 2008.

2.3.2 Consideraciones sobre la instalación

Cerrar otras aplicaciones
Dado que Communications Server interactúa con varios productos que podrían estar instalados en su sistema y requiere que se reinicie, debe cerrar otras aplicaciones antes de instalar Communications Server.

Ejecución automática del CD
Puede ejecutar esta funcionalidad en Windows XP, Windows Server 2003 (Standard Edition o Enterprise Edition), Windows Vista y Windows Server 2008.

Communications Server y Microsoft SNA Server
Communications Server y Microsoft SNA Server no se pueden instalar en la misma partición primaria. Muchos de los servicios comunes que coinciden entre los productos no son compatibles.

Derechos de usuario de administración necesarios
Un usuario de administración de Communications Server debe tener el derecho de usuario avanzado Cargar y descargar controladores de dispositivo, como se especifica en el Administrador de usuarios de Windows. Seleccione el menú Directivas > Derechos de usuario y debe estar seleccionado Mostrar derechos de usuario avanzados. Puede añadir explícitamente este derecho a cada usuario. También puede añadir este derecho de usuario al grupo de usuarios IBMCSADMIN.

Instalación del controlador de dominio
Al instalar Communications Server en un controlador de dominio primario o de copia de seguridad, debe habilitar el derecho de usuario Inicio de sesión local para los grupos de usuarios locales IBMCSADMIN y IBMCSAPI.

2.3.4 Mantenimiento del producto

Este Communications Server 6.4 es una instalación completa y sólo se pueden instalar los CSD para 6.4 después de esta instalación.

IBM proporciona un servicio de mantenimiento correctivo mediante:

  • arreglos individuales (arreglos APAR o APARs) y
  • revisiones del producto (CSD).

Disponibilidad de APAR y CSD

Arreglos APAR:

Communications Server incorpora arreglos APAR concretos para resolver defectos del producto. Un arreglo APAR sobre un componente concreto sustituirá a los APAR anteriores de ese componente. Los arreglos APAR también requieren que esté instalado el CSD más reciente.

CSD:

Un CSD es una renovación del producto con todos los APAR acumulativos. Se trata de una instalación completa pero instala el CSD de renovación en el mismo directorio y con las mismas características que el nivel anterior, después de que éste se haya desinstalado.

Encontrará los fixpacks más recientes y CSD para Communications Server para Windows bajo la marca de Websphere ubicada en Fix Central :

2.3.5 Habilitación del registro de anotaciones de Windows Installer
El registro de anotaciones de instalación de los productos de Communications Server se especifica cuando se invoca la instalación, ya sea a través del Launchpad o desde la línea de mandatos, especificando que desea realizar el registro de anotaciones de forma explícita.

Para Windows Installer 4.0 en Windows Vista y superiores, los productos de Communications Server tienen habilitado el registro de anotaciones verboso de forma predeterminada para las operaciones de instalación/desinstalación. Sin embargo, para los sistemas Windows anteriores a Vista, de forma predeterminada, la desinstalación iniciada desde la opción "Agregar o Quitar Programas" no se registrará.

En los sistemas más antiguos, es necesario añadir la siguiente entrada en el registro de la máquina para poder registrar la desinstalaciones iniciadas desde la opción "Agregar o Quitar Programas":

  • Clave del sistema = HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\Installer
  • Nombre del valor = Logging
  • Tipo de datos = REG_SZ (serie)
  • Datos = voicewarmup
Si desea más detalles, consulte el sitio web de registro "Cómo habilitar el registro de Windows Installer" en: http://support.microsoft.com/kb/223300/es

Para poder registrar una desinstalación, antes de desinstalar cualquiera de los tres productos de Communications Server ejecute msilogging_add.reg, que está instalado en Tools bajo el directorio de instalación principal de Communications Server, para actualizar el registro.

Nota: Puesto que el registro de Windows Installer se aplica a todos los productos que se instalan o desinstalan en el sistema, dejar este registro de anotaciones habilitado puede tener efectos negativos en el rendimiento y reducir el espacio en disco.

Después de la desinstalación, ejecute msilogging_remove.reg para eliminar la entrada. Aunque el producto se haya desinstalado, los archivos de msilogging permanecerán en Tools bajo el directorio de instalación principal de Communications Server. Estos archivo no se desinstalan con el producto.

Los registros de anotaciones de instalación estarán el directorio de registro que se haya especificado al instalar el producto. Cuando desintale el producto utilizando la opción "Agregar o Quitar Programas", el registro de anotaciones de desinstalación se encontrará en el directorio TEMP de la máquina. El nombre del archivo de registro de anotaciones tendrá el formato MSI***.log, donde *** es un número cualquiera. Utilice la indicación de fecha y hora para determinar qué archivos corresponden a la desinstalación de Communications Server.

5.1.1 Soporte IPv6 para TN3270E y cliente API SNA
El servidor TN3270E soporta los clientes IPv4 y IPv6 y proporciona el protocolo IPv6 instalado en la máquina del servidor. El protocolo IPv6 se puede instalar en Windows XP y en releases posteriores del sistema operativo Windows. El cliente API SNA de IBM también tiene habilitado el soporte IPv6.

NOTA: AnyNet SNA sobre IP no soporta el direccionamiento IPV6. La función AnyNet de Communications Server para Windows se va a anular en el próximo release. Debe planificar la sustitución de esta función con Enterprise Extender o el uso de una implementación de cliente API SNA.

5.1.2 Soporte para APPC Cancel Conversation
CANCEL_CONVERSATION es un verbo de control que cancela una conexión entre una LU local y una LU socia utilizando un programa de transacción específico (tp_id) y una conversación (conv_id).

La definición de la estructura VCB para el verbo CANCEL_CONVERSATION es la siguiente: 

typedef struct cancel_conversation

{

unsigned short opcode; /* código de operación del verbo */

unsigned char opext; /* código de extensión del verbo */

unsigned char format; /* formato */

unsigned short primary_rc; /* código de retorno primario */

unsigned long secondary_rc; /* código de retorno secundario */

unsigned char tp_id[8]; /* identificador de TP */

unsigned long conv_id; /* identificador de conversación */

} CANCEL_CONVERSATION;
Consulte la guía de programación de cliente/servidor para conocer el uso de este verbo. [Volver al principio de la subsección] [Volver al principio del documento] [Tabla de contenido]

5.1.3 Mejora de la sincronización del programa de transacción
La versión anterior de Communication Server para Windows sólo soporta tres niveles de sincronización (Alguno, Ninguno y Confirmar ). Pero se han añadido dos sincronizaciones más (SYNCPT negociable y SYNCPT necesario) a la definición del programa de transacción.

  • SYNCPT negociable El programa de transacción soporta el nivel de sincronización Ninguno, Confirmar o Punto de sincronismo.
  • SYNCPT necesario El programa de transacción soporta el nivel de sincronización Punto de sincronismo.

    El nivel de sincronización necesario se puede seleccionar al definir el programa de transacción desde el panel básico de la aplicación de configuración del nodo.

    5.1.5 ARB progresivo
    IBM Communications Server para Windows normalmente anuncia soporte en las conexiones RTP para todos los algoritmos ARB disponibles: estándar, modalidad de respuesta y modalidad progresiva. Para utilizar el proceso RTP normal a fin de que Communications Server para Windows utilice el mejor mecanismo RTP disponible según la capacidad del sistema remoto, establezca este parámetro en ANY.

    Para personalizar el funcionamiento de RTP, especifique uno de los siguientes valores para la palabra clave ARB_SUPPORT anterior de la sección NODO:

  • FORCE_STANDARD_ARB:
    Si se establece este valor, Communications Server para Windows sólo anunciará soporte para el algoritmo ARB estándar, no para el algoritmo de modalidad de respuesta o de modalidad progresiva.
  • NO_PROGRESSIVE_ARB:
    Si se establece este valor, Communications Server para Windows anunciará soporte para los algoritmos ARB estándar y de modalidad de respuesta, pero no para el algoritmo de modalidad progresiva.

    5.1.6 Retardo de conmutador de vía de acceso
    Retraso mínimo en segundos antes de que se produzca una conmutación de vía de acceso en las conexiones RTP. La especificación de un retraso evita intentos innecesarios de conmutación de vía de acceso ocasionados por la escasez temporal de recursos en el sistema remoto, en particular cuando no hay ninguna otra ruta disponible.

    El valor predeterminado para este parámetro es cero, lo que indica que puede producirse un intento de conmutación de vía de acceso en cuanto el protocolo indica que es necesaria.

    Consulte el capítulo 23, sección RTP_TUNNING, del manual de consulta de archivos de configuración para obtener más información.

    [Volver al principio de la subsección] [Volver al principio del documento] [Tabla de contenido]

    5.1.7 Inhabilitar soporte de activación remota para el enlace
    Se ha añadido un nuevo parámetro, DISABLE_REMOTE_ACT, al registro de definición de enlace. Evita que el nodo remoto active la estación de enlace. Los valores posibles son:
    • DISABLE_REMOTE_ACT=1, SÍ. Sólo el nodo local puede activar la estación de enlace; si el nodo remoto intenta activarla, CS Windows rechazará el intento.
    • DISABLE_REMOTE_ACT=0, NO (valor predeterminado). El nodo remoto puede activar la estación de enlace.

    El parámetro DISABLE_REMOTE_ACT del archivo .ACG forma parte de la definición de LINK_STATION. Este parámetro se puede establecer/anular en la definición de estación de enlace bajo el panel de reactivación.

Este programa de utilidad finaliza los ejecutable del cliente API SNA, sin necesidad de volver a arrancar el cliente.

    5.2.3 Configuración en Windows Terminal Client (WTC)
    No se requiere configuración especial en WTC. Inicie la sesión en WTS mediante el cliente de Escritorio remoto u otro software de WTC. Utilice la sesión configurada de Personal Communications definida en el WTS.

    La configuración en WTS es la siguiente:

    1. Configure el cliente API SNA para sesiones LUA. Esta configuración se puede efectuar para cada usuario.
      1. Seleccione Configuración INI local.
      2. Configure los datos globales como ID de usuario, Contraseña, etc. Para obtener más información, consulte la publicación Guía de iniciación rápida.
      3. Cree una definición LUA que incluya el nombre de la sesión LUA, la dirección IP del servidor, etc. Para obtener más información, consulte la publicación Guía de iniciación rápida.
      4. Guarde la configuración.
        • Si desea guardar esta configuración para un usuario concreto, cree un directorio con el nombre de ese usuario (por ejemplo, usuario1) y almacene el archivo de configuración con el nombre de ese usuario (por ejemplo, usuario1.ini). Para completar la configuración, vea el elemento número 3 debajo, "Para configurar un archivo .INI distinto para cada usuario."
        • Si desea que la configuración sea la misma para todos los usuarios, almacene el archivo en la ubicación por omisión con el nombre por omisión. El único cambio es el soporte de la configuración multiusuario.
    2. Configure Personal Communications para utilizar el cliente API SNA. El procedimiento es igual que el anterior, con o sin soporte WTS.
      1. Seleccione la interfaz cliente API.
      2. Seleccione la conexión LUA0,1,2,3 vía WINRUI.
      3. Pulse Parámetros de enlace y seleccione el nombre de la sesión configurada en el cliente API SNA.
      4. Mantenga los valores por omisión para el resto de los valores.
    3. Para configurar un archivo .INI distinto para cada usuario, haga lo siguiente:
      1. Inicie la sesión en el sistema como el usuario para el que necesita configurar el archivo .INI.
      2. Vaya al panel Propiedades del sistema > Opciones avanzadas y pulse Variables de entorno.
      3. Pulse Variables de usuario> Nueva.
      4. Escriba CSNTAPI como valor de la variable y entre la vía de acceso al archivo .INI.

        Por ejemplo, si la configuración del usuario 1 se encuentra en el archivo C:\usuario1\usuario1.ini y la configuración del usuario 2 se encuentra en C:\usuario2\usuario2.ini, la configuración es así:

        • Para el usuario 1, escriba CSNTAPI como valor de la variable C:\usuario1\usuario1.ini como vía de acceso
        • Para el usuario 2, escriba CSNTAPI como valor de la variable y C:\usuario2\usuario2.ini" como vía de acceso

    No hay más cambios en el procedimiento de configuración. La configuración de sesiones LUA, programas de transacciones, etc. permanece sin cambios. No es necesario cambiar la configuración de Personal Communications. Los pasos para configurar Personal Communications o cualquier aplicación LU siguen siendo los mismos.

    5.2.4 Notas sobre el rastreo y las anotaciones de mensajes del cliente API SNA
    • Si se selecciona la opción Todos los rastreos de API del Recurso de rastreo, los rastreos de depuración también quedarán registrados en el archivo pcatrace.dat. Cuando recopile rastreos de depuración, debe seleccionar la opción Todos los rastreos de API. Pulse Iniciar para empezar el rastreo. El rastreo de depuración se detendrá cuando el rastreo se haya detenido o cuando el usuario salga de la GUI del recurso de rastreo.
    • Las anotaciones de mensajes siempre se almacenan en el archivo MsgLog.dat. Los archivos de anotaciones de mensajes (MsgLog.dat) se colocarán en 2 archivos de 131 KB de tamaño cada uno. Esto permite reducir el tamaño del archivo de anotaciones de mensajes lo cual facilita su gestión. El recurso Infobundler recopilará los datos de ambos archivos de anotaciones de mensajes.
    • El rastreo lo puede iniciar un administrador y también un usuario que no sea administrador.
    • El rastreo detallado (pcatrace.dat) se recopilará en múltiples archivos. Por omisión, cada archivo tiene un tamaño de 100 K, y el número máximo de archivos es de 999.

      • Los valores predeterminados (tamaño de archivo y número máximo de archivos) pueden modificarse con el mandato tracedg /s TamañoArch /n NúmMáx. Este valor es global. El valor establecido por un usuario afectará a todos los usuarios.

      Por ejemplo, tracedg /s 92160 /n 20 creará cada archivo pcatrace con un tamaño de 90 k, y el número de archivos que se crearán es de 20. Después de 20 archivos, se sobregrabará el primer archivo.

    • La ubicación predeterminada para Msglog.dat y pcatrace.dat es el directorio especificado por "APPDATA" (este valor se recopilará por separado para cada usuario). Esto puede cambiarse estableciendo la variable de entorno CSNTAPITRCLOG.
    • Con esta versión se proporciona Infobundler para capturar los archivos de cliente clave necesarios para diagnosticar problemas. Siempre que intente resolver un problema con IBM, debería ejecutarlo y enviar la salida resultante. Para ejecutar Infobundler, vaya al directorio de instalación del cliente API de SNA (por ejemplo, C:\CSNTAPI) desde la línea de mandatos. Ejecute INFOB.exe desde la línea de mandatos. Por ejemplo:

      C:>INFOB

      Iniciará una serie de 13 pasos para finalmente recopilar la información (los 11 documentos mencionados a continuación) en el archivo cspd.exe, que se encuentra en el directorio de instalación (por omisión, C:\Archivos de programa\IBM\CS SNA API Client).

      Para resolver cualquier problema del que se informe, ejecute Infobundler y envíe el archivo cspd.exe al servicio de soporte de IBM.

      El resumen de características del recurso Infobundler para el cliente API SNA es el siguiente:

      1. Empaquetar MsgLog.dat (archivo(s) de anotaciones de mensajes de error)
      2. Empaquetar pcatrace.dat (archivo(s) de datos de rastreo de depuración)
      3. Empaquetar CSNTAPI.TRC (archivo de datos de rastreo sin formato)
      4. Empaquetar CSNTAPI.FMT (archivo de datos de rastreo con formato)
      5. Empaquetar APIERROR.LOG (archivo de anotaciones de error de API)
      6. Empaquetar Eventlogs
      7. Empaquetar Dr.Watson Log
      8. Empaquetar CSNTAPI.INI
      9. Empaquetar todas las claves del registro. Tenga en cuenta que las variables de entorno del usuario y las variables de entorno del sistema forman parte de la configuración del Registro. Las variables de entorno del usuario corresponderán a las del usuario que ejecuta Infobundler.
      10. Empaquetar el listado de directorio CSNTAPI
      11. Empaquetar el listado de directorio Windows\system32. Esto resultará útil para determinar si la aplicación está utilizando WINRUI32.DLL, WINAPPC32.DLL, etc. de la instalación del cliente API SNA o bien de la instalación de Microsoft Windows. Hemos observado que si se utilizan las API de la instalación de Microsoft Windows, la aplicación no funciona.

    5.2.5 Modificaciones del servidor de directorios LDAP
    La información de configuración del cliente API SNA se puede guardar en un directorio LDAP. Los servidores de directorios soportados son Netscape Directory Server Versión 4.0, IBM Directory Server Versión 3.1.1 y Lotus Domino Versión 5.0.

    Se proporcionan extensiones de esquema para Netscape Directory Server y deben añadirse a la configuración del servidor antes de configurar los clientes API SNA. Los otros servidores de directorios soportados ya contienen las definiciones de esquema necesarias. El control de acceso de entradas y atributos se debe controlar utilizando el programa de utilidad de administración del servidor de directorios.

    El programa de utilidad SNA API LDAP le permite modificar las entradas extendidas. El ID de usuario utilizado con este programa de utilidad debe tener acceso a las entradas de escritura del directorio. Puede utilizar el ID de administrador de directorios (que tiene estos permisos) al ejecutar el programa de utilidad de configuración del cliente.

    Par añadir las extensiones de esquema al servidor Netscape, añada las líneas siguientes al archivo slapd.conf, situado en el directorio \config del servidor de directorios de Netscape.

    include ibmcs-oc-ns.conf
    include ibmcs-at-ns.conf

    El nombre de la vía de acceso de este directorio depende del lugar de instalación del servidor de directorios y del nombre del servidor de directorios. Vea el ejemplo siguiente: 

    letra_unidad:\netscape\suitespot\slapd-nombre_sistema_principal\config

      6 Limitaciones

      6.1 SLP y niveles de cifrado distintos
      Una sesión cliente que acceda a un puerto TN3270 de alta seguridad con una consulta SLP podría conectarse a un puerto TN3270 de sólo autenticación, o viceversa, si ambos están configurados en el mismo servidor.

      Communications Server para Windows le permite configurar varios niveles de cifrado (Alto, Medio, Sólo autenticación) para una definición de puerto TN3270(E) o TN5250. Estos niveles de cifrado se encuentran en el panel Seguridad del diálogo Establecer definiciones de puerto TN3270.

      Algunos clientes TN3270 o TN5250 (como Host On-Demand Versión 5) no hacen un uso completo de la información de SSLv3 que Communications Server hace pública en un paquete SLP. Estos clientes se conectan al primer puerto con el nivel, ámbito, agrupación y carga de SSL que han solicitado, independientemente del nivel de cifrado. La conexión resultante podría ser una sesión que conecte con un puerto con un nivel de cifrado distinto al previsto.

      Para evitar esta situación, utilice sólo un nivel de cifrado, (Alto, Medio o Sólo autenticación) para cada máquina o ámbito. Sólo se puede acceder a los puertos de un servidor habilitando e inhabilitando la seguridad y la autenticación del cliente.

      6.2 Uso del cliente API SNA
      Los Clientes API SNA SNA son adecuados para entornos bifurcados en los que el número de usuarios que se conectan mediante el Cliente API SNA es bajo. Si realiza una consolidación de varias bifurcaciones en un centro de datos, o si utiliza una aplicación de servidor web, es recomendable utilizar un servidor en lugar de un Cliente API SNA.
